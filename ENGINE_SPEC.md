# ENGINE_SPEC.md

# Project Cascade Classic
## Engine Specification
### Version 1.0

---

# Назначение

Project Cascade Engine — модульный RTS-движок, разработанный специально для Project Cascade Classic.

Движок не является универсальным игровым движком.

Он оптимизирован исключительно под RTS.

Главные цели:

- высокая производительность;
- простота расширения;
- читаемый код;
- независимые игровые системы;
- возможность масштабирования без переписывания архитектуры.

---

# Основные принципы

Engine никогда не должен знать:

- какие существуют юниты;
- какие существуют здания;
- какие существуют фракции;
- какие существуют ресурсы.

Engine знает только:

- игровой цикл;
- время;
- ввод;
- камеру;
- сцену;
- обновление;
- рендер.

Игровая логика находится выше Engine.

---

# Архитектура

```
Browser

↓

Engine

↓

Game

↓

Systems

↓

Entities

↓

Renderer
```

Engine управляет жизненным циклом игры.

Game управляет игровыми правилами.

Systems реализуют механики.

Renderer только отображает происходящее.

---

# Игровой цикл

Используется Fixed Timestep.

```
60 UPS
```

Logic обновляется всегда с одинаковой скоростью.

Render не ограничивается.

```
Update

↓

Physics

↓

AI

↓

Economy

↓

Combat

↓

Animation

↓

Render
```

---

# Главный класс

Engine

Отвечает только за:

- запуск;
- остановку;
- игровой цикл;
- время;
- FPS;
- Update();
- Render();

Никакой игровой логики.

---

# Game

Game является владельцем мира.

Он хранит:

- карту;
- игроков;
- системы;
- сущности.

Game ничего не рисует.

---

# Renderer

Renderer получает только данные.

Он никогда не изменяет игру.

Renderer отвечает за:

- terrain;
- units;
- buildings;
- effects;
- ui;
- minimap.

---

# Camera

Camera хранит:

- x
- y
- zoom

Camera ничего не знает о юнитах.

Она лишь преобразует координаты.

---

# Input

InputManager отвечает за:

- мышь;
- клавиатуру;
- горячие клавиши;
- drag selection;
- правый клик.

Input не должен отдавать приказы юнитам напрямую.

Он создаёт события.

---

# Event System

Все взаимодействия происходят через события.

Например

```
LeftMouseDown

SelectionChanged

BuildingStarted

UnitCreated

AttackOrder

MoveOrder
```

Это уменьшает связанность систем.

---

# Entity System

Все игровые объекты являются Entity.

Entity содержит:

- id
- position
- owner
- hp
- maxHp
- state

Entity не знает как себя рисовать.

---

# Unit

Наследуется от Entity.

Добавляет:

- скорость;
- оружие;
- обзор;
- команды;
- очередь приказов.

---

# Building

Наследуется от Entity.

Добавляет:

- строительство;
- производство;
- очередь производства;
- энергопотребление.

---

# Projectile

Отдельный класс.

Пули не являются частью юнита.

Это позволяет:

- ракеты;
- снаряды;
- лазеры;
- трассеры;
- артиллерию.

---

# Selection System

Отвечает за:

- выделение;
- группы;
- drag box;
- ctrl groups.

Не содержит UI.

---

# Combat System

Отвечает за:

- атаку;
- броню;
- урон;
- смерть;
- уничтожение.

Не содержит эффектов.

---

# Economy System

Отвечает за:

- нефть;
- сталь;
- расходы;
- производство ресурсов.

---

# Construction System

Отвечает за:

- строительство;
- размещение зданий;
- сетку строительства.

---

# Production System

Отвечает за:

- создание юнитов;
- очереди;
- производство зданий.

---

# Fog Of War

Отдельная система.

Renderer лишь отображает результат.

---

# Pathfinding

Отдельная система.

Никакой логики движения внутри Unit.

Unit лишь запрашивает маршрут.

---

# AI

AI работает через те же команды, что и игрок.

AI не может "телепортировать" юнитов.

---

# UI

UI никогда не управляет игрой напрямую.

UI отправляет команды в Game.

---

# Assets

Все ресурсы загружаются через AssetManager.

Запрещается:

```
new Image()
```

в игровых системах.

---

# Save System

В будущем сохраняются только игровые данные.

Engine не сериализуется.

---

# Запрещено

Нельзя:

- использовать глобальные переменные;
- смешивать Render и Logic;
- обращаться Renderer → Game;
- изменять Game из Renderer;
- использовать setInterval();
- использовать setTimeout() для игровой логики;
- создавать God Objects;
- хранить несколько крупных систем в одном файле.

---

# Требования к производительности

Classic должна стабильно работать:

500+ юнитов

200+ зданий

100+ снарядов

60 UPS

на среднем игровом ПК.

---

# Правило разработки

Каждая новая система проходит три этапа:

1. Specification
2. Implementation
3. Testing

Ни одна система не добавляется без предварительной спецификации.

---

# Главный принцип движка

Engine должен быть максимально "глупым".

Вся игровая логика должна находиться вне Engine.

Чем меньше Engine знает об игре, тем проще поддерживать проект и добавлять новые возможности.
